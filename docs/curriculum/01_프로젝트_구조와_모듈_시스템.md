# 01. 프로젝트 구조와 모듈 시스템

## 이 문서에서 배우는 것

- Cargo Workspace가 뭔지, 왜 쓰는지
- `mod`, `pub`, `use`의 동작 원리
- DMaster가 3개의 crate으로 나뉜 이유

---

## 1. Cargo Workspace

### 개념

하나의 저장소에 여러 개의 Rust 패키지(crate)를 묶는 구조.

```
DMaster/
├── Cargo.toml          ← Workspace root (빌드 대상 선언)
├── dmaster_core/       ← 라이브러리 crate
├── dmaster_gui/        ← 바이너리 crate (GUI)
└── dmaster_cli/        ← 바이너리 crate (CLI)
```

**Workspace root의 Cargo.toml:**

```toml
[workspace]
members = ["dmaster_core", "dmaster_gui", "dmaster_cli"]
name = "dmaster_workspace"
```

`members`에 나열된 디렉토리가 각각 독립 crate이지만, 빌드 결과물(`target/`)은 공유한다.

### 왜 Workspace를 쓰는가?

```
┌─────────────────────────────────────────────────┐
│                    Workspace                     │
│                                                  │
│  ┌──────────────┐                                │
│  │ dmaster_core │◀──────────┬──────────┐        │
│  │ (라이브러리)   │           │          │        │
│  └──────────────┘           │          │        │
│                      의존        의존       │
│                             │          │        │
│                    ┌────────┴──┐  ┌────┴─────┐  │
│                    │dmaster_gui│  │dmaster_cli│  │
│                    │ (GUI exe) │  │ (CLI exe) │  │
│                    └───────────┘  └──────────┘  │
│                                                  │
│  공유: target/, Cargo.lock                        │
└─────────────────────────────────────────────────┘
```

| 장점 | 설명 |
|------|------|
| **코드 재사용** | `dmaster_core`를 GUI와 CLI 모두에서 사용 |
| **빌드 캐시 공유** | 의존성을 한 번만 컴파일 |
| **일관된 버전** | `Cargo.lock` 하나로 모든 crate의 의존성 버전 통일 |
| **관심사 분리** | 로직(core) ↔ UI(gui) ↔ 인터페이스(cli) 분리 |

### 각 crate의 역할

**dmaster_core** (lib crate):
```toml
[lib]
path = "src/lib.rs"    # 라이브러리 진입점
```
- 디스플레이 조회, 프로필 저장/적용의 **모든 로직**
- 플랫폼 독립적인 인터페이스 제공
- GUI든 CLI든 이 crate만 호출하면 동작

**dmaster_gui** (bin crate):
```toml
[dependencies]
slint = "1.15"
dmaster_core = { path = "../dmaster_core" }   # 로컬 경로로 의존
```
- Slint UI 렌더링
- 사용자 입력 → `dmaster_core` 함수 호출 → 결과 표시

**dmaster_cli** (bin crate):
```toml
[dependencies]
dmaster_core = { path = "../dmaster_core" }
```
- 터미널 메뉴 인터페이스
- 같은 core 로직을 텍스트 기반으로 제공

📌 **핵심**: `path = "../dmaster_core"` — 로컬 경로 의존성. 원격 crates.io 대신 같은 Workspace 내 crate을 직접 참조한다.

## 2. 모듈 시스템 (mod)

### lib.rs — 라이브러리의 루트

```rust
// dmaster_core/src/lib.rs

pub mod apply;           // src/apply.rs를 모듈로 선언
pub mod backend;         // src/backend/ 디렉토리를 모듈로 선언
pub mod display_info;    // src/display_info.rs를 모듈로 선언
pub mod profile;         // src/profile.rs를 모듈로 선언
pub mod query;           // src/query.rs를 모듈로 선언
```

`pub mod X;`는 두 가지 일을 한다:
1. `X`라는 이름의 모듈을 **선언**
2. `pub`이므로 외부 crate에서 **접근 가능**

### 파일 ↔ 모듈 매핑

```
mod 선언              →  실제 파일 위치
──────────────────────────────────────
pub mod apply;        →  src/apply.rs
pub mod query;        →  src/query.rs
pub mod profile;      →  src/profile.rs
pub mod display_info; →  src/display_info.rs
pub mod backend;      →  src/backend/mod.rs   ← 디렉토리면 mod.rs
```

⚠️ **주의**: `backend`은 디렉토리이므로, `src/backend/mod.rs`가 해당 모듈의 루트가 된다.

### backend/mod.rs — 하위 모듈 관리

```rust
// dmaster_core/src/backend/mod.rs

pub trait DisplayBackend { /* ... */ }

#[cfg(target_os = "windows")]
mod windows;                    // 비공개 모듈 (pub 없음)

#[cfg(target_os = "linux")]
mod linux;

#[cfg(target_os = "linux")]
mod gnome_wayland;

#[cfg(not(any(target_os = "windows", target_os = "linux")))]
mod unsupported;
```

여기서 `mod windows;`에는 `pub`이 **없다**. 즉:
- `backend` 모듈 내부에서만 `windows::WindowsDisplayBackend`에 접근 가능
- 외부에서는 `get_backend()` 함수를 통해서만 사용

이것이 **캡슐화**다. 구현 상세를 숨기고 인터페이스만 노출한다.

## 3. Visibility (공개 범위)

### pub의 단계

```
키워드              접근 가능 범위
────────────────────────────────────
(없음)              같은 모듈 내에서만
pub(crate)          같은 crate 내에서만
pub(super)          부모 모듈에서만
pub                 어디서든 (외부 crate 포함)
```

### DMaster에서의 실전 사용

```rust
// display_info.rs — 모든 필드가 pub
pub struct DisplayConfig {
    pub label: Option<String>,     // pub: GUI에서 직접 읽어야 하므로
    pub device_name: String,
    pub width: u32,
    // ...
}
```

```rust
// backend/mod.rs
mod windows;   // pub 없음: 내부 구현은 숨김
mod linux;     // pub 없음

pub fn get_backend() -> Box<dyn DisplayBackend> {  // pub: 외부에서 호출
    // ...
}
```

```rust
// profile.rs
fn sanitize_profile_name(name: &str) -> String {  // pub 없음: 내부 헬퍼
    // ...
}

pub fn save_profile(/* ... */) -> Result<PathBuf, String> {  // pub: API
    // ...
}
```

📌 **원칙**: 외부에 필요한 것만 `pub`. 나머지는 비공개가 기본.

## 4. Re-export (pub use)

### 문제: 경로가 너무 깊다

re-export 없이 GUI에서 함수를 쓰려면:
```rust
use dmaster_core::profile::save_profile;
use dmaster_core::profile::load_profiles;
use dmaster_core::query::get_display_profile;
use dmaster_core::apply::apply_profile;
use dmaster_core::display_info::DisplayMapping;
```

### 해결: lib.rs에서 re-export

```rust
// dmaster_core/src/lib.rs

pub use apply::{apply_profile, apply_profile_with_mapping};
pub use display_info::*;                    // 모든 public 항목을 re-export
pub use profile::{
    delete_profile, load_profile_by_name, load_profiles, 
    profiles_dir, save_profile, ProfileInfo,
};
pub use query::get_display_profile;
```

이제 GUI에서:
```rust
use dmaster_core::{
    apply_profile, get_display_profile, load_profiles,
    save_profile, DisplayMapping, DisplayTopology,
};
```

깔끔하게 한 줄로 모든 것을 가져올 수 있다.

### Re-export 흐름도

```
외부 crate (dmaster_gui)
    │
    │  use dmaster_core::save_profile;
    │
    ▼
lib.rs
    │  pub use profile::save_profile;
    │
    ▼
profile.rs
    │  pub fn save_profile(...) { ... }
    │
    ▼
실제 구현
```

💡 **DMaster 설계 포인트**: `lib.rs`가 "공식 API 목록" 역할을 한다. 여기서 re-export되지 않은 것은 내부 구현으로 간주.

## 5. 빌드 시스템

### cargo 명령어

```bash
# Workspace 전체 빌드
cargo build

# 특정 crate만 빌드
cargo build -p dmaster_gui
cargo build -p dmaster_cli

# Release 빌드 (최적화)
cargo build --release -p dmaster_gui

# 크로스 컴파일 (Linux → Windows)
cargo build --release --target x86_64-pc-windows-gnu -p dmaster_gui
```

### build.rs — 빌드 스크립트

```rust
// dmaster_gui/build.rs
fn main() {
    slint_build::compile("ui/app.slint").unwrap();
}
```

`build.rs`는 컴파일 **전에** 실행되는 스크립트:
1. `cargo build` 실행
2. `build.rs` 먼저 실행 → `.slint` 파일을 Rust 코드로 컴파일
3. 생성된 Rust 코드 + `src/main.rs`를 함께 컴파일

이래서 `main.rs`에서 `slint::include_modules!()` 매크로로 Slint 컴포넌트를 사용할 수 있다.

## 정리: DMaster의 모듈 구조 전체도

```
dmaster_core (lib)
│
├── lib.rs ─────────── re-export: 공식 API 선언
│
├── display_info.rs ── 데이터 타입: DisplayConfig, DisplayProfile, ...
│
├── query.rs ───────── get_display_profile() → backend 위임
│
├── apply.rs ───────── apply_profile() → backend 위임
│
├── profile.rs ─────── save/load/delete → 파일 시스템 I/O
│
└── backend/
    ├── mod.rs ─────── trait DisplayBackend + get_backend() 팩토리
    ├── windows.rs ─── #[cfg(windows)] WinAPI 구현
    ├── linux.rs ───── #[cfg(linux)] xrandr 구현
    ├── gnome_wayland.rs ── #[cfg(linux)] Mutter D-Bus 구현
    └── unsupported.rs ──── 폴백 (에러 반환)
```

## 확인 문제

1. `dmaster_gui`에서 `dmaster_core::backend::linux::LinuxDisplayBackend`에 직접 접근할 수 있는가? 왜 안 되는가?
2. `pub mod`과 `mod`의 차이는?
3. `lib.rs`에서 `pub use display_info::*;`가 하는 일은?
4. `build.rs`는 언제 실행되는가?

<details>
<summary>정답</summary>

1. **접근 불가**. `linux` 모듈에 `pub`이 없으므로 `backend` 모듈 외부에서 접근할 수 없다. `get_backend()` 함수를 통해서만 사용 가능.
2. `pub mod`은 외부 crate에서도 해당 모듈에 접근 가능. `mod`만 쓰면 같은 crate 내에서만 접근 가능.
3. `display_info` 모듈의 모든 `pub` 항목(struct, enum, fn 등)을 `dmaster_core` 루트에서 직접 접근 가능하게 만든다. `dmaster_core::DisplayConfig`처럼 쓸 수 있게 됨.
4. `cargo build` 시 소스 코드 컴파일 **전에** 실행된다. Slint 파일을 Rust 코드로 변환하는 데 사용.
</details>
