# 03. 데이터 모델 설계

## 이 문서에서 배우는 것

- `display_info.rs`의 struct/enum 설계 의도
- Derive 매크로의 동작 원리
- 타입 간 변환 패턴 (`MonitorProfile` ↔ `DisplayProfile`)

---

## 1. 전체 타입 관계도

```
MonitorProfile (저장용 - JSON 파일)
├── schema_version: String         "0.2.0"
├── name: String                   "home_dual"
├── description: Option<String>    Some("집 듀얼 모니터 설정")
├── created_at: String             "1709900000"
├── topology: DisplayTopology      Extend
└── displays: Vec<DisplayConfig>
    ├── [0] DisplayConfig          메인 모니터
    └── [1] DisplayConfig          서브 모니터

DisplayProfile (런타임용 - 메모리)
├── topology: DisplayTopology
└── displays: Vec<DisplayConfig>

DisplayMapping (매핑 정보)
├── current_display_name: String   "DP-1"
└── profile_display_index: usize   0
```

핵심 관계:

```
MonitorProfile ──── .to_display_profile() ────▶ DisplayProfile
                                                     │
DisplayProfile ──── MonitorProfile::new() ◀──────────┘
                    (이름, 설명, 시간 추가)
```

## 2. DisplayConfig — 모니터 하나의 정보

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DisplayConfig {
    pub label: Option<String>,     // 사람이 읽는 이름: "DELL U2720Q"
    pub device_name: String,       // OS 식별자: "\\.\DISPLAY1" 또는 "DP-1"
    pub device_id: String,         // 하드웨어 ID
    pub device_key: String,        // 레지스트리 키 (Windows) / 모드 ID (Wayland)
    pub width: u32,                // 해상도 가로
    pub height: u32,               // 해상도 세로
    pub position_x: i32,           // 멀티모니터 위치 X (음수 가능)
    pub position_y: i32,           // 멀티모니터 위치 Y (음수 가능)
    pub orientation: u32,          // 0=Normal, 1=90°, 2=180°, 3=270°
}
```

### 왜 `position`은 `i32`인가?

멀티모니터에서 왼쪽 모니터의 좌표는 음수가 될 수 있다:

```
  ┌──────────┐ ┌──────────┐
  │ 모니터 2   │ │ 모니터 1   │
  │ (-1920,0) │ │ (0, 0)   │
  └──────────┘ └──────────┘
```

`u32`(unsigned)면 음수 표현 불가 → `i32`(signed) 사용.

### 왜 `label`은 `Option<String>`인가?

- Windows: 항상 device_name으로 채울 수 있음
- Linux (xrandr): 모델명을 가져올 수 없는 경우가 있음
- `None`이면 `device_name`을 대신 표시

## 3. DisplayTopology — Enum 설계

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum DisplayTopology {
    Extend,          // 확장 디스플레이
    Clone,           // 미러링
    Internal,        // 내장 모니터만
    External,        // 외부 모니터만
    Unknown(i32),    // 미지의 토폴로지 (에러 코드 포함)
}
```

### Enum이 Rust에서 강력한 이유

C/Java의 enum은 정수 나열일 뿐이지만, Rust의 enum은 **각 variant가 데이터를 가질 수 있다**:

```rust
Unknown(i32)   // Unknown variant는 i32 값을 하나 담고 있음
```

이렇게 하면 "알 수 없는 토폴로지"일 때도 원래 코드 값을 보존한다:

```rust
// windows.rs에서:
fn query_display_topology() -> Result<DisplayTopology, String> {
    let status = unsafe { GetDisplayConfigBufferSizes(...) };
    if status != 0 {
        return Ok(DisplayTopology::Unknown(status));  // 에러 코드를 보존
    }
    // ...
}
```

### 💡 match와 함께 사용

```rust
// gui/src/main.rs
fn topology_label(t: &DisplayTopology) -> &'static str {
    match t {
        DisplayTopology::Extend => "Extended",
        DisplayTopology::Clone => "Cloned",
        DisplayTopology::Internal => "Internal Only",
        DisplayTopology::External => "External Only",
        DisplayTopology::Unknown(_) => "Unknown",    // i32 값은 무시
    }
}
```

`match`는 모든 variant를 처리해야 한다 (**exhaustive**). 하나라도 빠뜨리면 컴파일 에러.

## 4. MonitorProfile — 저장 전용 래퍼

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MonitorProfile {
    pub schema_version: String,
    pub name: String,
    pub description: Option<String>,
    pub created_at: String,
    pub topology: DisplayTopology,
    pub displays: Vec<DisplayConfig>,
}
```

### DisplayProfile과의 차이

```
DisplayProfile (런타임)         MonitorProfile (저장)
─────────────────────          ─────────────────────
topology                       schema_version  ← 추가
displays                       name            ← 추가
                               description     ← 추가
                               created_at      ← 추가
                               topology
                               displays
```

왜 두 개로 나눴을까?

- **DisplayProfile**: OS에서 조회한 "지금 화면 상태". 이름이나 설명 같은 건 없다.
- **MonitorProfile**: 사용자가 저장한 "프로필". 메타데이터(이름, 설명, 생성일)가 추가.

### 타입 변환 메서드

```rust
impl MonitorProfile {
    // DisplayProfile → MonitorProfile (메타데이터 추가)
    pub fn new(
        name: String,
        description: Option<String>,
        created_at: String,
        profile: DisplayProfile,        // 소유권을 가져옴
    ) -> Self {
        Self {
            schema_version: String::from("0.2.0"),
            name,
            description,
            created_at,
            topology: profile.topology,     // DisplayProfile을 분해
            displays: profile.displays,     // 필드를 각각 꺼냄
        }
    }

    // MonitorProfile → DisplayProfile (메타데이터 제거)
    pub fn to_display_profile(&self) -> DisplayProfile {
        DisplayProfile {
            topology: self.topology.clone(),    // &self이므로 clone 필요
            displays: self.displays.clone(),
        }
    }
}
```

📌 **`new`에서는 move**, **`to_display_profile`에서는 clone** — 왜?

- `new`: `profile: DisplayProfile`을 소유권으로 받아서, 필드를 꺼내(move) struct에 넣음. 원본 `DisplayProfile`은 소멸.
- `to_display_profile`: `&self`로 빌려 읽으므로, 값을 꺼낼 수 없음. `.clone()`으로 복사본 생성.

## 5. Derive 매크로

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
```

이 한 줄이 자동 생성하는 코드:

| Derive | 생성되는 것 | 용도 |
|--------|------------|------|
| `Debug` | `fmt::Debug` trait | `println!("{:?}", value)` 디버그 출력 |
| `Clone` | `.clone()` 메서드 | 깊은 복사 |
| `Serialize` | serde 직렬화 | struct → JSON |
| `Deserialize` | serde 역직렬화 | JSON → struct |

`Debug` 없이 `println!("{:?}", profile)` 하면 컴파일 에러.
`Clone` 없이 `profile.clone()` 하면 컴파일 에러.
`Serialize` 없이 `serde_json::to_string(&profile)` 하면 컴파일 에러.

### Derive가 가능한 조건

struct의 **모든 필드**가 해당 trait를 구현해야 한다:

```rust
pub struct DisplayConfig {
    pub label: Option<String>,   // Option<String>: Debug ✓ Clone ✓ Serialize ✓
    pub width: u32,              // u32: Debug ✓ Clone ✓ Serialize ✓
    // 모든 필드가 Debug, Clone, Serialize를 구현하므로
    // DisplayConfig도 derive 가능
}
```

## 6. DisplayMapping — 간단하지만 중요한 구조체

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DisplayMapping {
    pub current_display_name: String,    // 현재 연결된 모니터: "DP-1"
    pub profile_display_index: usize,    // 프로필에서 몇 번째 설정을 적용할지: 0
}
```

이 struct가 존재하는 이유:

```
저장할 때 (집)              적용할 때 (회사)
───────────────            ───────────────
DP-1: 2560x1440            HDMI-1: ???
HDMI-1: 1920x1080          DP-2: ???
```

모니터 커넥터 이름이 달라지면 어떤 설정을 어디에 적용할지 모른다.
`DisplayMapping`이 "현재 HDMI-1에 프로필의 0번 설정을 적용해라"라고 매핑해준다.

## 정리: 타입 설계 원칙

| 원칙 | DMaster 적용 |
|------|-------------|
| **값이 없을 수 있으면 `Option`** | `label`, `description` |
| **실패할 수 있으면 `Result`** | 모든 public 함수의 반환 타입 |
| **경우의 수가 정해져 있으면 `enum`** | `DisplayTopology` |
| **데이터 묶음이면 `struct`** | `DisplayConfig`, `MonitorProfile` |
| **용도가 다르면 별도 타입** | `DisplayProfile` (런타임) vs `MonitorProfile` (저장) |

## 확인 문제

1. `position_x`가 `u32`가 아닌 `i32`인 이유는?
2. `DisplayTopology::Unknown(i32)`에서 `(i32)`는 어떤 역할인가?
3. `MonitorProfile::new()`에서 `profile.topology`가 move되는 이유는?
4. `#[derive(Serialize)]`가 작동하려면 어떤 조건이 필요한가?

<details>
<summary>정답</summary>

1. 멀티모니터 배치에서 왼쪽 모니터의 X 좌표가 음수(`-1920`)가 될 수 있으므로.
2. `Unknown` variant가 정수 데이터를 하나 포함한다는 의미. WinAPI에서 반환된 에러 상태 코드를 보존.
3. `profile` 매개변수가 `DisplayProfile` (소유권)으로 전달되었으므로, 내부 필드를 move로 꺼낼 수 있다. `&DisplayProfile`이었다면 clone이 필요.
4. struct의 모든 필드 타입이 `Serialize` trait를 구현해야 한다. `String`, `u32`, `i32`, `Option<T>`, `Vec<T>`, 사용자 정의 타입(역시 Serialize 필요) 등.
</details>
