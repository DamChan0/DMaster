# 02. 소유권과 타입 시스템

## 이 문서에서 배우는 것

- Ownership, Borrowing, Lifetime — DMaster 코드에서의 실전 사례
- `String` vs `&str`, `Vec<T>` vs `&[T]`
- `Option<T>`, `Result<T, E>` 활용 패턴

---

## 1. Ownership (소유권)

### 원칙

Rust에서 모든 값은 **딱 하나의 변수(소유자)**에 귀속된다. 소유자가 스코프를 벗어나면 값은 자동으로 해제(drop)된다.

```rust
fn main() {
    let name = String::from("home_setup");  // name이 소유자
    save_something(name);                    // name의 소유권이 함수로 이동 (move)
    // println!("{name}");                   // 컴파일 에러! name은 이미 move됨
}
```

### 💡 DMaster에서의 Move

`profile.rs`의 `save_profile`:

```rust
pub fn save_profile(
    name: &str,              // 빌려옴 (borrow) — 소유권 안 가져감
    description: Option<String>,  // 소유권 이동 (move) — 호출자가 더 이상 못 씀
    profile: &DisplayProfile,     // 빌려옴
) -> Result<PathBuf, String> {
```

왜 `name`은 `&str`이고 `description`은 `Option<String>`인가?

- `name`: 읽기만 하면 됨 → 빌려서 사용
- `description`: `MonitorProfile` 안에 저장해야 함 → 소유권을 가져와서 struct에 넣음

```rust
let stored_profile = MonitorProfile::new(
    name.to_string(),     // &str → String 변환 (새 String 생성)
    description,          // Option<String>을 그대로 move
    current_timestamp_string(),
    profile.clone(),      // 참조를 clone해서 새 복사본 생성
);
```

## 2. Borrowing (빌림)

### `&T` (불변 참조) vs `&mut T` (가변 참조)

```
┌─────────────────────────────────────────────────┐
│  규칙:                                           │
│  • &T는 동시에 여러 개 가능                        │
│  • &mut T는 한 번에 하나만 가능                     │
│  • &T와 &mut T는 동시에 존재할 수 없음              │
└─────────────────────────────────────────────────┘
```

### 💡 DMaster에서의 Borrowing 패턴

**패턴 1: 읽기 전용 빌림 (`&self`, `&T`)**

```rust
// backend/mod.rs
pub trait DisplayBackend {
    fn get_display_profile(&self) -> Result<DisplayProfile, String>;
    fn apply_profile(&self, profile: &DisplayProfile) -> Result<(), String>;
}
```

`&self`: 자기 자신을 불변으로 빌림. Backend 구조체의 상태를 변경하지 않음.
`&DisplayProfile`: 프로필 데이터를 읽기만 함.

**패턴 2: 슬라이스 빌림 (`&[T]`)**

```rust
// apply.rs
pub fn apply_profile_with_mapping(
    profile: &DisplayProfile,
    mappings: &[DisplayMapping],    // Vec<DisplayMapping>의 슬라이스
) -> Result<(), String> {
```

`&[DisplayMapping]`은 `Vec<DisplayMapping>`을 빌려 읽는 것. 호출 측:

```rust
let mappings = vec![DisplayMapping { ... }, DisplayMapping { ... }];
apply_profile_with_mapping(&profile, &mappings);  // &Vec<T> → &[T] 자동 변환
// mappings는 여전히 사용 가능 (소유권 유지)
```

**패턴 3: Iterator에서의 빌림**

```rust
// profile.rs — load_profiles()
for entry in entries {
    let path = entry.path();
    let content = fs::read_to_string(&path)?;     // &path: Path를 빌려서 읽기
    let profile = parse_profile_content(&path, &content)?;
    profiles.push(ProfileInfo {
        name: profile.name.clone(),                // clone: 빌린 데이터에서 복사
        path: path.to_string_lossy().to_string(),
        profile,                                   // move: profile 소유권 이동
    });
}
```

📌 **`.clone()` vs move**: `profile.name`은 `MonitorProfile` 안의 `String`. `profile`을 통째로 `ProfileInfo`에 넣을 건데, `name`도 따로 필요하니까 먼저 `.clone()`으로 복사해 둔다.

## 3. String vs &str

### 차이

```
String                          &str
──────────────────────         ──────────────────────
• 소유권 있음 (heap 할당)         • 빌림 (다른 String의 일부)
• 수정 가능                      • 읽기 전용
• 크기 가변                      • 고정 뷰
• .to_string(), format!()       • "리터럴", &String, .as_str()
```

### 💡 DMaster에서의 사용 패턴

```rust
// 함수 인자: 읽기만 할 때 → &str
pub fn save_profile(name: &str, ...) { }
pub fn delete_profile(name: &str) { }

// struct 필드: 소유해야 할 때 → String
pub struct DisplayConfig {
    pub device_name: String,
    pub device_id: String,
}

// 변환: &str → String
name.to_string()           // 가장 일반적
String::from(name)         // 동일한 효과
format!("{}x{}", w, h)    // 포맷팅하면서 String 생성
```

⚠️ **흔한 실수**: 함수가 `&str`을 받는데 `String`을 넘기면?
```rust
let name = String::from("test");
save_profile(&name, ...);          // &String → &str 자동 변환 (deref coercion)
save_profile("literal", ...);      // &str 리터럴도 OK
```

## 4. Lifetime

### 개념

빌린 참조가 얼마나 오래 유효한지 컴파일러에게 알려주는 표시.

### 💡 DMaster에서의 Lifetime: `select_profile`

```rust
// dmaster_cli/src/main.rs
fn select_profile<'a>(
    profiles: &'a [dmaster_core::ProfileInfo],
) -> Option<&'a dmaster_core::ProfileInfo> {
```

이 함수 시그니처를 분해하면:

```
fn select_profile<'a>(             // 'a라는 라이프타임 매개변수 선언
    profiles: &'a [ProfileInfo],   // 입력: 'a만큼 유효한 슬라이스 참조
) -> Option<&'a ProfileInfo>       // 출력: 입력과 같은 'a만큼 유효한 참조
```

**의미**: "반환되는 `&ProfileInfo`는 입력 `profiles`가 살아있는 동안만 유효하다"

```rust
fn apply_profile_flow(use_mapping: bool) {
    let profiles = load_profiles().unwrap();       // profiles 생성
    let selected = select_profile(&profiles);       // profiles를 빌려서 하나 선택
    // selected는 profiles가 살아있으므로 여기서 사용 가능
    if let Some(p) = selected {
        apply_profile(&p.profile.to_display_profile());
    }
}   // profiles가 여기서 drop → selected도 무효화 (이미 스코프 밖)
```

📌 **대부분의 경우 라이프타임을 직접 쓸 필요 없다**. 컴파일러가 자동 추론(lifetime elision)한다. DMaster에서 명시적 라이프타임이 필요한 곳은 `select_profile` 하나뿐.

## 5. Option\<T\>

### 개념

값이 있을 수도 없을 수도 있는 타입. null 대신 사용.

```rust
enum Option<T> {
    Some(T),    // 값 있음
    None,       // 값 없음
}
```

### 💡 DMaster에서의 Option 사용

**필드에서: "선택적 데이터"**

```rust
pub struct DisplayConfig {
    pub label: Option<String>,        // 디스플레이 이름이 없을 수 있음
    // ...
}

pub struct MonitorProfile {
    pub description: Option<String>,  // 설명 없이 저장할 수 있음
    // ...
}
```

**처리 패턴들:**

```rust
// 1. unwrap_or_default — None이면 기본값
p.profile.description.as_deref().unwrap_or_default()
//   Some("내 설정") → "내 설정"
//   None            → ""

// 2. unwrap_or — None이면 지정값
d.label.as_deref().unwrap_or(d.device_name.as_str())
//   Some("DELL U2720Q") → "DELL U2720Q"
//   None                → "\\.\DISPLAY1"

// 3. or_else — None이면 다른 Option 시도
source_display.label.clone().or_else(|| current_display.label.clone())
//   source에 label 있으면 그거, 없으면 current의 label 시도

// 4. map — Some이면 변환, None이면 그대로
std::env::var_os("HOME")
    .or_else(|| std::env::var_os("USERPROFILE"))
    .map(PathBuf::from)
    .ok_or_else(|| String::from("HOME/USERPROFILE is not set"))
```

## 6. Result\<T, E\>

### 개념

성공 또는 실패를 표현. 예외(exception) 대신 사용.

```rust
enum Result<T, E> {
    Ok(T),      // 성공 + 결과값
    Err(E),     // 실패 + 에러 정보
}
```

### 💡 DMaster의 Error Handling 전략

DMaster는 `Result<T, String>`을 일관되게 사용:

```rust
pub fn save_profile(...) -> Result<PathBuf, String> { }
pub fn load_profiles() -> Result<Vec<ProfileInfo>, String> { }
pub fn get_display_profile() -> Result<DisplayProfile, String> { }
pub fn apply_profile(...) -> Result<(), String> { }
```

에러 타입이 `String`인 이유: 간단한 프로젝트에서 커스텀 에러 타입 없이 에러 메시지만 전달.

**`map_err` 체이닝 — DMaster의 핵심 패턴:**

```rust
// profile.rs
fs::create_dir_all(&dir).map_err(|error| {
    format!("failed to create profile directory '{}': {error}", dir.display())
})?;
```

풀어쓰면:
```rust
// 위의 한 줄은 이것과 같다:
let result = fs::create_dir_all(&dir);
match result {
    Ok(()) => (),  // 성공하면 계속 진행
    Err(io_error) => {
        let message = format!("failed to create ...: {io_error}");
        return Err(message);  // 에러 메시지와 함께 즉시 반환
    }
}
```

**`?` 연산자**: `Result`가 `Err`이면 현재 함수에서 즉시 반환.

```rust
pub fn load_profile_by_name(name: &str) -> Result<ProfileInfo, String> {
    let profiles = load_profiles()?;   // Err면 여기서 바로 return Err(...)
    profiles
        .into_iter()
        .find(|profile| profile.name == name)
        .ok_or_else(|| format!("profile '{name}' was not found"))
}
```

## 7. Clone과 Copy

### 차이

```
Copy                              Clone
──────────────────────           ──────────────────────
• 비트 단위 복사 (저렴)            • .clone() 명시 호출 (비용 있음)
• 스택 데이터만 (u32, bool, ...)  • 힙 데이터 포함 (String, Vec, ...)
• move 안 됨 (항상 복사)           • move가 기본, clone은 명시적
```

### 💡 DMaster에서의 Clone

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct DisplayProfile {
    pub topology: DisplayTopology,
    pub displays: Vec<DisplayConfig>,
}
```

`#[derive(Clone)]`으로 `.clone()` 메서드가 자동 생성됨.

```rust
// save_profile에서:
let stored_profile = MonitorProfile::new(
    name.to_string(),
    description,
    current_timestamp_string(),
    profile.clone(),    // &DisplayProfile을 복사해서 소유권 확보
);
```

`profile`은 `&DisplayProfile` (빌린 참조)인데, `MonitorProfile`에 넣으려면 소유권이 필요. 그래서 `.clone()`으로 깊은 복사를 한다.

## 정리: DMaster에서 볼 수 있는 소유권 패턴

| 상황 | 패턴 | DMaster 예시 |
|------|------|-------------|
| 읽기만 할 때 | `&T`, `&str`, `&[T]` | `fn save_profile(name: &str, ...)` |
| 소유해야 할 때 | `T`, `String`, `Vec<T>` | struct 필드: `pub name: String` |
| 있을 수도 없을 수도 | `Option<T>` | `pub label: Option<String>` |
| 성공/실패 | `Result<T, E>` | `-> Result<PathBuf, String>` |
| 빌린 데이터를 저장 | `.clone()` | `profile.clone()` |
| &str → String | `.to_string()` | `name.to_string()` |
| 에러 변환 | `.map_err()` + `?` | `fs::write(&path, json).map_err(...)? ` |

## 확인 문제

1. `save_profile(name: &str, description: Option<String>, ...)`에서 `name`은 왜 `&str`이고 `description`은 왜 `String`인가?
2. `profile.clone()`을 안 쓰면 어떤 에러가 나는가?
3. `?` 연산자가 하는 일을 `match`로 풀어서 설명하라.
4. `as_deref().unwrap_or_default()`는 `Option<String>`에서 어떤 변환 과정을 거치는가?

<details>
<summary>정답</summary>

1. `name`은 읽기만 하면 되므로 빌림(`&str`). `description`은 `MonitorProfile` struct에 저장해야 하므로 소유권(`String`)을 가져온다.
2. `cannot move out of *profile which is behind a shared reference` — `&DisplayProfile` (빌린 참조)에서 값을 꺼내(move) 올 수 없다는 에러.
3. `let x = expr?;` → `let x = match expr { Ok(v) => v, Err(e) => return Err(e.into()) };`
4. `Option<String>` → `.as_deref()` → `Option<&str>` → `.unwrap_or_default()` → `&str`. `Some(String)` → `Some(&str)` → `&str 값`, `None` → `""`.
</details>
